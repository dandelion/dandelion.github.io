Building an asset graph

You have seen in the introduction about asset bundles that you can connect bundles together with the dependencies property. Now let's see in detail how works the dependency mechanism.

Bundle hierarchy

Building a bundle hierarchy is as simple as using the parent property.

Take the example of DataTables, which is a jQuery plugin. Its bundle representation is: 

{
    "bundle" : "datatables",
    "parent": "jquery",
    "assets": [
       {
          "name": "datatables-js",
          "version": "1.9.4",
          "type": "js",
          "locations": {
             "cdn": "//ajax.aspnetcdn.com/ajax/jquery.dataTables/1.9.4/jquery.dataTables.js"
          }
       }, {
          "name": "datatables-css",
          "version": "1.9.4",
          "type": "css",
          "locations": {
             "cdn": "//ajax.aspnetcdn.com/ajax/jquery.dataTables/1.9.4/css/jquery.dataTables.css"
          }
       }
    ]
}

In the above bundle, jquery is set as a parent bundle of datatables .

For a better overview, take a look at the bundle hierarchy:

Keep in mind that by creating bundles, you actually create a tree-like structure of assets.

There are many benefits to manipulate bundles:

  • thanks to the bundle hierarchy, all assets are ordered. This order is extremely important since Dandelion will preserve this order when injecting <script> and <link> tags into the HTML code.
  • it provides a layer of abstraction on top of all your web resources, providing you a single entry point for managing your assets.
  • you can easily load assets in pages by requesting a bundle. Read more about how you can interact with the asset stack.
  • thanks to this layer of abstraction, we'll provide in future versions some tooling allowing you to debug your asset stack or even to fully integrate the asset in the build process. Stay tuned!

Once bundles created, you have now an asset stack!

  We chose the term asset stack in reference to the term technology stack, which is traditionnaly used to define a list of technologies used to provide a software solution or application. In Dandelion, asset stack actually means the set of client-side dependencies to be used in your application.

Redefining bundles

Once created, a bundle is not "static". Actually you can redefine bundles in several ways:

  • You can overload some assets defined in a bundle (lien)
  • You can also completely override some assets (lien)

It is worth noting that whether it's at the moment of the asset stack building or when it is accessed, Dandelion always apply a set of rules. Read more about the storage rules.

Vendor bundles

The dandelion-core JAR ships what we call vendor bundles, i.e. vendor librairies that you can already use without writing the JSON files by yourself.

The versions of these librairies are set arbitrarily but you can of course modify them by redefining the bundles as mentioned above.
