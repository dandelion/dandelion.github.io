Introducing asset loaders

As we saw in the asset stack introduction, all scopes are stored in JSON files. This section describes how these files are loaded into the system.

Standard asset loader

The DandelionAssetJsonLoader is the main loader used by default by Dandelion.

Its role is to scan the classpath recursively, starting from the /dandelion folder, for files that ends with .json .

All files found in the /dandelion folder will then be converted into an internal representation: AssetComponent (lien javadoc).

For example, in the following Maven structure, the standard asset loader will pick up and load scope1.json , scope2.json and scope3.json files. 

YourWebApp
|__ src
   |__ main
      |__ resources
         |__ dandelion
            |__ scope1.json
            |__ scope2.json
            |__ scope3.json

Vendor asset loader

This is the second and last asset loader to be defaultly enabled by Dandelion.

This loader is intended to load vendor's librairies, such as jQuery or Bootstrap. The only difference with the other built-in loader is that it scans for JSON files inside the /vendor folder and all subfolders.

For example, some vendor scopes are bundled with the dandelion-core JAR. They're all located under the /vendor folder. 

dandelion-core
|__ src
   |__ main
      |__ resources
         |__ vendor
            |__ jquery.json
            |__ jquery2.json
            |__ bootstap2.json
            |__ bootstap3.json
            |__ ...

Of course, you are encouraged to follow the same practice, for a better asset organization.

  Except the location, there is currently no difference with the standard asset loader. In upcoming releases however, this will allow us to easily apply a specific caching policy.

How it works with multiple active asset loaders?

All active asset loaders are mutually exclusive. It means that a folder or subfolder will never be scanned twice.

This rule ensures maximum consistency with the loaded assets.
